## ADDED Requirements

### Requirement: Turma-Restricted Availability
The system SHALL allow professors to restrict recurrence rules to specific turmas, so that only students enrolled in those turmas can see and book the generated slots.

#### Scenario: Professor creates recurrence restricted to turmas
- **WHEN** a professor creates or edits a recurrence rule
- **THEN** the system SHALL present an optional multi-select of active turmas from the tenant
- **AND** if turmas are selected, the rule is linked to those turmas via `agendamento_recorrencia_turmas`
- **AND** if no turmas are selected, the rule remains visible to all students (default behavior)

#### Scenario: Student views slots filtered by turma enrollment
- **WHEN** a student requests available slots via `getAvailableSlots`
- **THEN** the system SHALL fetch the student's turma enrollments from `alunos_turmas`
- **AND** for each recurrence rule with turma restrictions, only include it if the student is enrolled in at least one linked turma
- **AND** for recurrence rules without turma restrictions, include them for all students

#### Scenario: Student browses professors filtered by turma
- **WHEN** a student views the professor list via `getProfessoresDisponiveis`
- **THEN** the system SHALL filter each professor's recurrence rules by the student's turma enrollments
- **AND** a professor with only turma-restricted rules that don't match the student's turmas SHALL NOT appear in the list

#### Scenario: Calendar availability respects turma filter
- **WHEN** `getAvailabilityForMonth` is called for a student
- **THEN** day-level availability indicators SHALL only reflect slots the student has access to based on turma enrollment

#### Scenario: Booking validation enforces turma restriction
- **WHEN** a student attempts to book a slot generated by a turma-restricted recurrence rule
- **THEN** the system SHALL verify the student is enrolled in at least one of the linked turmas
- **AND** if the student is not enrolled, the system SHALL reject with "O professor não tem disponibilidade neste horário."

#### Scenario: Recurrence without turma restriction maintains universal access
- **WHEN** a recurrence rule has no entries in `agendamento_recorrencia_turmas`
- **THEN** all students in the tenant SHALL see the generated slots
- **AND** the behavior SHALL be identical to the current system

#### Scenario: Professor views restricted rules with turma indicator
- **WHEN** a professor views their recurrence rules in `RecorrenciaManager`
- **THEN** rules with turma restrictions SHALL display a visual indicator showing linked turma names
- **AND** rules without restrictions SHALL show no indicator

#### Scenario: Turma deactivation affects slot visibility
- **WHEN** a turma linked to a recurrence rule is deactivated (`ativo = false`)
- **THEN** students formerly in that turma SHALL no longer see slots from that rule (unless enrolled in another linked active turma)
- **AND** the recurrence rule SHALL NOT become universally visible

## MODIFIED Requirements

### Requirement: Recurrence-based Availability
The system SHALL use `agendamento_recorrencia` table as the single source of truth for professor availability, with optional turma-based visibility restrictions.

#### Scenario: Professor configures weekly availability
- **WHEN** a professor accesses the availability management page
- **THEN** they can create recurrence patterns with:
  - Service type (plantao/mentoria)
  - Start and end dates
  - Day of week
  - Start and end times
  - Slot duration (15, 30, 45, or 60 minutes)
  - Turma restriction (optional multi-select of active turmas)

#### Scenario: System generates available slots
- **WHEN** `getAvailableSlots` is called for a professor and date
- **THEN** slots are generated based on `agendamento_recorrencia` entries
- **AND** if a student context is provided, turma-restricted rules are filtered by enrollment
- **AND** existing appointments and bloqueios are excluded
- **AND** minimum advance time is respected

#### Scenario: Legacy table is deprecated
- **WHEN** any code references `agendamento_disponibilidade`
- **THEN** a deprecation warning should be logged
- **AND** the operation should still work for backwards compatibility

### Requirement: Professor Selection
The system SHALL allow students to browse and select available professors before scheduling an appointment, respecting turma-based availability restrictions.

#### Scenario: Student views available professors
- **WHEN** a student navigates to `/agendamentos`
- **THEN** a list of professors with available slots is displayed
- **AND** each professor card shows name, photo, and next available slot
- **AND** only slots accessible to the student (based on turma enrollment) are considered

#### Scenario: Student filters professors
- **WHEN** a student enters a search term
- **THEN** the professor list is filtered by name or specialty

#### Scenario: Student selects professor
- **WHEN** a student clicks on a professor card
- **THEN** the student is navigated to `/agendamentos/[professorId]`
- **AND** the scheduling calendar is displayed for that professor
